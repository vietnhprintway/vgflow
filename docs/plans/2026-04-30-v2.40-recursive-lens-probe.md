# Plan v2.40.0 — Mandatory Recursive Lens Probe + Goal Back-Flow + Phase 0 Diagnostic

## Context

### Smoke test reverses architecture decision

User initially confirmed "subprocess gemini không drive được MCP playwright". Live smoke test (run 30/04/2026) chứng minh ngược lại:

| Test | Model | Kết quả |
|---|---|---|
| `gemini -p "navigate + snapshot example.com" -m gemini-2.5-pro --allowed-mcp-server-names playwright1` | Pro 2.5 | **PASS** — trả về "Example Domain" |
| Same prompt, `-m gemini-2.5-flash --allowed-mcp-server-names playwright2` | Flash 2.5 | **PASS** — trả về "Example Domain" |

→ Architecture subprocess + MCP playwright **không broken**. v3.0.0 rewrite KHÔNG cần. Vấn đề user gặp ở production phase là **localized bug** ở:
- Kit prompt thiếu động lực → LLM không gọi browser tools
- base_url loading regex sai
- Auth fixture tokens.local.yaml shape mismatch

Plan: **giữ scope recursive lens probe (v2.40), thêm Phase 0 Diagnostic để tìm root cause**. Phase 0 phải pass trước khi ship recursive layer (vì recursive layer build trên cùng spawn pattern; nếu pattern broken trong production, recursive cũng fail).

## Phase 0: Diagnostic (BLOCKER — phải làm trước recursive layer)

### Mục đích

Tìm chính xác chỗ Phase 2d "đi qua nhưng không drive browser" trong production. Smoke test pass nhưng user dogfood phase fail → có bug localized ở 1 trong 7 hypotheses.

### Procedure

1. **User cung cấp 1 phase thật từng fail** (PrintwayV3 hoặc tương tự, có CRUD-SURFACES.md + tokens.local.yaml)

2. **Add `--debug` flag vào `scripts/spawn-crud-roundtrip.py`** (~30 LOC):
   - Capture FULL stdout (không chỉ tail 2000 chars)
   - Capture FULL stderr
   - Log subprocess env vars resolved (GEMINI_API_KEY presence, không log value)
   - Log resolved prompt (full content fed to gemini, redacted secrets)
   - Log resolved kit_text length, context_block JSON keys
   - Write debug log to `phase_dir/runs/.debug-{run_id}.log`

3. **Run dry-run + actual spawn** với phase fixture:
   ```bash
   python scripts/spawn-crud-roundtrip.py --phase-dir <fixture> --dry-run --json
   python scripts/spawn-crud-roundtrip.py --phase-dir <fixture> --resource <one> --role admin --debug
   ```

4. **Inspect 3 things**:
   - `runs/<run_id>.json` — run artifact: có `network_log[]` entries không? Steps[] có `evidence_ref` không?
   - `runs/.debug-{run_id}.log` — subprocess stdout/stderr full text
   - `gemini -p` exit code

5. **Diagnose root cause** dựa vào 1 trong 7 hypothesis:

| # | Hypothesis | Symptom | Fix |
|---|---|---|---|
| 1 | base_url load regex sai | Worker đọc base_url=None → không navigate | Tighten regex hoặc switch sang YAML/MD parser |
| 2 | tokens.local.yaml shape mismatch | auth_token trong context_block là null hoặc string không phải dict với token field | Schema validation + clearer error message |
| 3 | Kit prompt không actionable | LLM phản hồi text-only "I would navigate to X..." không gọi tool | Tighten prompt: "MUST call browser_navigate as first action" |
| 4 | CRUD-SURFACES.md thiếu field | spawn loop builds plan trống → 0 spawns | Validator surface required fields trước spawn |
| 5 | Run artifact path Permission denied | Worker LLM ghi không được, returns success nhưng file empty | Pre-check + chmod + atomic write |
| 6 | Stdout truncated | LLM ghi log dài, subprocess capture 2000 char tail thiếu evidence | Increase capture limit + tee to file |
| 7 | tokens.local.yaml file missing | context_block.auth_token = null silently | Pre-check tồn tại, fail loud |

6. **Apply fix**, re-run cùng phase fixture, verify run artifact có `network_log` thật.

### Deliverable Phase 0

- 1 commit: `--debug` flag added + (likely) 1-3 small fixes per hypothesis triggered
- Documentation: `docs/diagnostic-2026-04-30.md` (ngoài scope plan mode, viết sau khi exit) — what was wrong, root cause, fix applied
- Verified: Phase 2d run artifact với phase fixture có actual browser interactions

Phase 0 is a **gate** — recursive lens probe (Phase 1+) only ships AFTER Phase 0 passes.

## Phase 1: Recursive Lens Probe (post-Phase-0)

### Activation rule (6-rule eligibility check)

```
ELIGIBILITY (must check, then decide):
   1. phase_profile IN ("feature", "feature-legacy", "hotfix")
   2. surface SET includes "ui" OR "ui-mobile"
   3. CRUD-SURFACES.md declares >=1 resource
   4. touched_resources INTERSECT crud_resources != ∅
      (computed from build SUMMARY.md + git diff vs phase start)
   5. NOT (visual-only profile / a11y profile / read-only dashboard)
   6. ENV-CONTRACT.md (v2.39) declares disposable seed data
      AND third_party_stubs all set to "stubbed"

IF all 6 pass → activate, run probe
IF any fail → emit explicit no-op evidence into PHASE_DIR:
   .recursive-probe-skipped.yaml { reasons: [...], at: <ISO> }
   This is REQUIRED — no silent skip.
   Override only via --skip-recursive-probe="<reason>" (OVERRIDE-DEBT critical).
```

### Mode caps (no `off` mode)

| Mode | Depth cap | Lens application | Worker count guard | Goal cap |
|---|---|---|---|---|
| `light` (default) | 2 | All ship-ready lenses on action buttons + form triggers + tabs | ~15 workers/phase | 50 |
| `deep` | 3 | All lenses on every clickable + recursive sub-views | ~40 workers/phase | 150 |
| `exhaustive` | 4 | All lenses + state probe + cross-resource cascade | ~100 workers/phase | 400 |

### Probe execution mode (orthogonal với mode caps trên)

`/vg:review {phase} --recursion={light|deep|exhaustive} --probe-mode={auto|manual|hybrid}`:

| `--probe-mode` | Behavior | Khi dùng |
|---|---|---|
| `auto` (default) | VG spawns Gemini Flash subprocess workers như hiện tại. Tự động end-to-end, không cần user can thiệp. | Phase chạy CI/CD, dogfood thông thường, cost OK |
| `manual` | VG **không spawn** worker. Thay vào đó generate `recursive-prompts/MANIFEST.md` + per-probe prompt files. User mở CLI ngoài (gemini interactive, codex interactive, hoặc Claude Code trong VS Code), paste prompt, CLI session tự run MCP playwright + ghi run artifact JSON về `runs/`. VG sau đó verify run artifacts xuất hiện đầy đủ. | Phase chạy environment không support subprocess (sandbox restriction), user muốn control cost theo từng probe, debug từng lens probe riêng, hoặc test prompt quality |
| `hybrid` | VG spawn auto cho lenses confidence cao (CRUD round-trip, authz-negative); generate manual prompts cho các lens phức tạp/expensive (business-logic, ssrf chain). User paste những phần này. Một số quan trọng nhất user kiểm soát. | Phase quan trọng cần human-in-the-loop cho complex lens; CI/CD chạy auto, sensitive phase chạy hybrid |

#### Manual mode trade-offs

- **Pros**: zero subprocess dependency, user picks any CLI, debuggable per-probe, cost transparent
- **Cons**: user time-cost cao (paste 18 prompts), no parallelism, no termination guard enforcement (user can run forever), no findings-broker (prompts độc lập)
- **Mitigation**: MANIFEST.md ghi rõ termination rules + paste budget cap. `verify-manual-run-artifacts.py` enforce schema sau session.

### Goal back-flow (dedupe + aggregate)

Naive "1 stub per clickable per depth" → goal explosion. Replace với canonical dedupe + behavior-class aggregation:

1. Workers write run artifact `runs/<tool>/recursive-{element}-d{N}.json` where `<tool> ∈ {codex, gemini, claude}` (per-tool subdir để tránh overwrite khi cross-tool validation; existing v2.35 schema + replay v2.39 + new `worker_tool` field)
2. Workers write per-worker `goals-{worker_id}.partial.yaml` (ephemeral)
3. After all workers complete, manager runs `scripts/aggregate-recursive-goals.py`:
   - Canonical key: `{view, stable_selector_or_role, action_semantic, lens, resource, assertion_type}`
   - 1 goal per behavior class (not per DOM element)
   - Per-mode cap (light=50, deep=150, exhaustive=400)
   - Overflow → `recursive-goals-overflow.json` (raw, không vào TEST-GOALS-DISCOVERED.md)
4. Single-writer atomic merge into `TEST-GOALS-DISCOVERED.md`

Goal stub schema (granular fields):

```yaml
id: G-RECURSE-{behavior_class_hash}
source: review.recursive_probe
depth: 2
lens: authz_negative_path
view: "/admin/topup-requests"
element_class: row_action      # mutation_button | form_trigger | tab | row_action | sub_view_link | bulk_action
selector_hash: a3f8c2d1
resource: topup_requests
parent_goal_id: G-CRUD-topup-requests-list-admin
priority: critical
```

### 8 Termination guards

1. **Visited-state hash** — `(view_url_canonical, role, lens, element_selector_hash)` → skip if hit
2. **URL canonicalization** — strip volatile params, normalize → canonical form
3. **Selector/action memoization** — within single worker session, same `(stable_selector, action_semantic)` max 1 probe
4. **Mutation budget** — max 50 actual POST/PUT/PATCH/DELETE per phase across all workers; beyond → workers downgrade Read-only
5. **Wall-clock timeout** — per-worker 5min hard, per-phase recursive layer 30min hard
6. **Artifact size cap** — per-worker run artifact JSON max 500KB, beyond → truncate + telemetry
7. **Per-resource idempotency** — same `(resource × role × lens)` max 1 worker per depth level
8. **Diminishing returns** — 0 new behavior-class goals last round → halt

All 8 active by default. Per-mode override only relaxes wall-clock or budget, never disables hash/canonicalization/memoization.

| **NO** code fix recommendations | Build phase implements fixes |
| **NO** severity assignment | Re-review verifies |
| **NO** "exploit chains to" reasoning | (existing v2.37 + v2.39 pipeline) |

## How to explore recursively (anti-script discipline)

- Click element, capture network response baseline
- After each action, browser_snapshot. If new buttons/forms/sub-views/modals appear → click those too (recursive)
- If probe yields anomaly (unexpected status code, peer data leak, state-machine bypass, ...) → dig deeper:
  - Replay request với different role/token
  - Modify request body fields
  - Check if anomaly affects neighbor records
- DO NOT follow a fixed click sequence. Adapt to what you observe.
- DO NOT skip "boring-looking" elements without at least 1 click — coverage > narrow-focus.

# Lens: IDOR (Object-Level Authorization)

## Threat model
Endpoint exposes object ID (path/query/body) and uses it to fetch/mutate state. If server doesn't verify caller owns object, attacker can swap ID → access/modify other users' data. White-box review: capture object ID from baseline call, replay với swap (peer-tenant ID, sequential ID, deleted ID, admin-only ID), assert 403/404 NOT 200.

## Activation context (auto-injected)
- View: `${VIEW_PATH}`
- Element: `${ELEMENT_DESCRIPTION}` (selector: `${SELECTOR}`)
- Resource: `${RESOURCE}`
- Role: `${ROLE}` (token: `${TOKEN_REF}`)
- Peer token (other tenant, same role): `${PEER_TOKEN_REF}`
- Base URL: `${BASE_URL}`
- Output: `${OUTPUT_PATH}`
- Action budget: 12

## Reconnaissance

1. browser_navigate(`${BASE_URL}${VIEW_PATH}`) với `Authorization: Bearer ${TOKEN}`
2. browser_snapshot, locate `${SELECTOR}`, click + capture baseline `OBJ_ID_SELF` from response

## Objective

Find IDOR/BOLA on `${ELEMENT}` and any sub-elements that emerge. Object IDs are everywhere — paths, query strings, JSON bodies, JWT claims, GraphQL args. Probe horizontal access (peer with same role), vertical access (admin-only objects), batch operations (bulk arrays), and pagination/cursor leaks.

## Probe ideas (combine freely)

- Capture object IDs from baseline → replay request với `${PEER_TOKEN_REF}` (peer tenant) → check if peer accessed your data (status 200 returning `OBJ_ID_SELF`)
- Sequential ID guess: `OBJ_ID_SELF ± 1`, `± 100`, `± 1000` → check if response leaks neighbor records (cross-check `ownerId`/`tenantId` fields)
- For bulk endpoints: inject peer ID mid-array → check if bulk processes mixed-tenant IDs without rejection
- For admin-only objects (look for "admin", "staff", "_internal" in URL/payload): attempt access as non-admin role
- Check expansion params (`?include=`, `?expand=`, `?fields=`) — IDOR often hides in projection bypass
- For CSV/PDF/export endpoints: leak object IDs via parameter swap
- Background jobs (`/jobs/<id>/result`): IDs sequential, swap → leak others' job output
- For pagination cursors: try `cursor=<peer's cursor>` to skip filter and pull other tenants' pages
- After mutation, check if response leaks related entities (siblings/parent IDs you shouldn't see)

## How to explore recursively

- Click element, capture baseline. Note all IDs in URLs/responses.
- After click, snapshot → if new clickables/modals appear, probe those too (sub-elements)
- If you find anomaly (peer data leaked OR cross-owner mutation succeeded) → DIG:
  - Try same probe with different roles to find blast radius
  - Try mutation (PUT/PATCH/DELETE) on the leaked ID
  - Check audit log endpoint to see if your action was logged
- DO NOT follow scripted A/B/C/D order. Adapt to evidence.
- Action budget 12 — typically 2-3 probes per anomaly chain
